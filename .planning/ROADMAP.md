# Roadmap: Oddscope

**Created:** 2026-03-20
**Core Value:** 予測市場の「今何が起きているか」を、日本語で瞬時に把握できること
**Granularity:** Standard (6 phases)

## Phases

- [ ] **Phase 1: データ基盤・基本表示** - Polymarketデータを日本語で表示する基盤構築
- [ ] **Phase 2: テーマ整理・ノイズ抑制** - テーマ別整理とノイズ抑制で情報を見つけやすく
- [ ] **Phase 3: 急変検知・分析** - 急変検知とAI分析で「何が・なぜ動いたか」を把握
- [ ] **Phase 4: Discord通知** - リアルタイム通知で急変を逃さない
- [ ] **Phase 5: 朝チェック体験** - 朝のまとめダッシュボードで昨晩の動きを把握
- [ ] **Phase 6: 収益化** - フリーミアムモデルで継続的な収益化

## Phase Details

### Phase 1: データ基盤・基本表示
**Goal**: Polymarket市場データを日本語で一覧表示でき、ユーザーが「何の市場か」を一瞬で理解できる状態を作る

**Depends on**: なし (基盤フェーズ)

**Requirements**: MKTD-01, MKTD-02, MKTD-04

**Success Criteria** (what must be TRUE):
  1. ユーザーはPolymarket市場の一覧を日本語で閲覧できる
  2. 市場タイトルと説明文が日本語翻訳され、「何の市場か」が一瞬で理解できる
  3. 曖昧な市場・米政治/規制ネタに文脈補足が表示され、日本人が「それ何?」とならない
  4. カテゴリ・出来高・変動率でフィルタリング・ソートができる

**Plans:** 3 plans

Plans:
- [ ] 01-01-PLAN.md — プロジェクトスキャフォールディング + DB + 型定義 + テスト基盤
- [ ] 01-02-PLAN.md — データ同期パイプライン（翻訳・テーマ分類・Cron Job）
- [ ] 01-03-PLAN.md — 市場一覧UI（テーブル・フィルタ・レスポンシブ）

### Phase 2: テーマ整理・ノイズ抑制
**Goal**: テーマ別整理とノイズ抑制により、ユーザーが関心のある情報を素早く見つけられ、低流動性市場のノイズに邪魔されない状態を作る

**Depends on**: Phase 1 (データ基盤が動作している前提)

**Requirements**: MKTD-03, MKTD-05, MKTD-06, MKTD-07, SHAR-01, SHAR-02

**Success Criteria** (what must be TRUE):
  1. 市場がテーマ別(米政治、AI、暗号資産、地政学、マクロ等)にグルーピングされて表示される
  2. 低流動性市場が自動的に抑制され、重要度スコアで優先表示される市場が分かる
  3. 同種市場が束ねて表示され、重複ノイズが解消される
  4. テーマ軸で横断検索ができ、関連市場を素早く発見できる
  5. 市場・テーマ単位のパーマリンクで共有でき、X等でネタを拾いやすい
  6. データは共通市場モデルで管理され、将来的なマルチソース拡張に備えている

### Phase 3: 急変検知・分析
**Goal**: 急変市場を自動検知し、「何が動いたか」「なぜ動いたか」をAI分析で把握できる状態を作る

**Depends on**: Phase 1 (データ基盤), Phase 2 (テーマ軸での検知が必要)

**Requirements**: ANLY-01, ANLY-02, ANLY-03, ANLY-04

**Success Criteria** (what must be TRUE):
  1. 急変市場(1時間で±15%等)が自動抽出され、ハイライト表示される
  2. テーマ単位でも急変検知ができ、関連市場群の動きを把握できる
  3. 「なぜ動いたか」のAI下書きが生成され、手動編集後に表示される
  4. 原因と断定せず「関連ニュース」として提示され、自信が低い場合は無理に説明しない
  5. 価格変動とニュースのタイミング差が表示され、ユーザーが因果関係を判断できる
  6. コピペしやすい要約フォーマットで、X発信・記事ネタとして使える

### Phase 4: Discord通知
**Goal**: Discord通知で急変を配信し、ユーザーがリアルタイムで重要な動きを逃さない状態を作る

**Depends on**: Phase 3 (急変検知)

**Requirements**: NOTF-01, NOTF-02, NOTF-03

**Success Criteria** (what must be TRUE):
  1. Discord急変通知が日本語で自然に届き、一目で「見る価値あるか」判断できる
  2. 通知にテーマ情報と関連ニュース1行が含まれ、文脈が理解できる
  3. 通知の間引き設計により、ノイズにならない頻度で配信される

### Phase 5: 朝チェック体験
**Goal**: 朝のまとめダッシュボードで、昨晩の主要な動きをテーマごとに整理して把握できる状態を作る

**Depends on**: Phase 3 (急変検知データ), Phase 4 (Discord基盤)

**Requirements**: MORN-01, MORN-02, MORN-03, NOTF-04

**Success Criteria** (what must be TRUE):
  1. 朝ダイジェストが主画面として機能し、昨晩の急変をまとめて把握できる
  2. テーマごとに「何が起きたか」が整理表示され、関心領域を素早くチェックできる
  3. 通知から開いた時も朝見た時もUI・情報が破綻せず、一貫した体験が提供される
  4. 朝のまとめダイジェストがDiscordにも配信され、Web以外でも閲覧できる

**Plans**: TBD

### Phase 6: 収益化
**Goal**: フリーミアムモデルで無料ユーザーを集客し、有料ユーザーから継続的な収益を得られる状態を作る

**Depends on**: Phase 1-5 (全機能が動作し、価値が実証されている前提)

**Requirements**: MONT-01, MONT-02

**Success Criteria** (what must be TRUE):
  1. 無料プラン(一覧＋遅延データ)と有料プラン(通知＋分析＋リアルタイム)が実装され、境界が明確
  2. ユーザー認証・課金基盤が動作し、有料登録・決済フローが完結する
  3. 無料ユーザーにプレミアム機能の「チラ見せ」があり、有料プランへの導線が明確
  4. レート制限・データ遅延がプラン別に適用され、価値の差が体感できる

## Critical Path

**Phase 1の成功が全ての前提:**
- Polymarket API統合の安定性 (WebSocket接続断対策が必須)
- 日本語翻訳の品質 (ドメイン辞書実装)
- 共通市場モデルの設計 (将来拡張を見据えた抽象化)

**Phase 2でプロダクトの中心軸が確立:**
- テーマ別整理が全機能(急変検知・朝まとめ・通知)に効く
- ノイズ抑制が「見る価値あるか」判断の鍵

**Phase 3で差別化が実現:**
- 急変検知 + AI分析がコア価値
- LLMコスト管理が収益化の前提

**Phase 4でDiscord通知が稼働:**
- Discord通知のノイズ制御が利用継続の鍵

**Phase 5で朝チェック体験が完成:**
- Web朝ダイジェスト + Discord朝まとめの両チャネル完成

**Phase 6で収益化実現:**
- Phase 1-5の価値実証なしでは課金困難
