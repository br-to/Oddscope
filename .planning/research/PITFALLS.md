# Pitfalls Research

**Domain:** 予測市場インテリジェンスプラットフォーム
**Researched:** 2026-03-20
**Confidence:** MEDIUM-HIGH

## Critical Pitfalls

### Pitfall 1: Polymarket WebSocket接続の静的な崩壊

**What goes wrong:**
WebSocket接続が表面的には維持されている（readyState: OPEN）が、20-30分後にデータストリームが停止する。アプリケーションは正常動作していると判断するが、実際には市場の急変を検知できなくなる。

**Why it happens:**
Polymarketの`real-time-data-client`は内部的なハートビート/ping-pong機構が不完全で、サーバー側の無音タイムアウトを検知できない。また、クライアントライブラリは自動再接続ロジックを実装していない（Issue #14の報告通り、メンテナンス状態も不明確）。

**How to avoid:**
1. WebSocket接続に独自のハートビート層を実装（30秒ごとにping送信、pong未受信で再接続）
2. データストリームのメッセージ頻度を監視（市場活動時間帯に5分間メッセージなし→異常）
3. 接続断を前提としたエクスポネンシャルバックオフ再接続ロジック（初回1秒、最大60秒）
4. フォールバック: WebSocket停止時にHTTP polling（REST API）で代替

**Warning signs:**
- ログに`onMessage`コールバック実行が5分以上記録されない
- 市場活動時間帯（米営業時間）にデータ更新が止まる
- `readyState`はOPENだがメッセージキューが空のまま
- 特定市場の最終更新タイムスタンプが30分以上前

**Phase to address:**
Phase 1（データ取得基盤）で必須実装。MVP段階で発生すると信頼性が致命的に低下。

### Pitfall 2: Polymarket CLOB APIの認証遅延トラップ

**What goes wrong:**
API認証キーを作成直後、約2分間は401 Unauthorizedエラーが返される。デプロイ直後やキーローテーション時に、システムが「認証が壊れている」と誤判断し、不必要なロールバックやアラートが発生する。

**Why it happens:**
CLOBバックエンドの認証情報伝播に遅延がある（Issue #311報告済み）。ドキュメントに記載なし。開発者はキー作成→即座に使用を想定するが、実際には内部キャッシュ更新待ちが必要。

**How to avoid:**
1. API認証キー作成後、2-3分のwarm-up期間を設ける
2. 認証エラー時に即座に失敗とせず、指数バックオフリトライ（最大5分）
3. デプロイフローにキー作成→待機→検証のステップを組み込む
4. 監視: 401エラーが3分以上継続した場合のみアラート

**Warning signs:**
- デプロイ直後に短時間の401エラーが集中
- キーローテーション後の一時的なAPI障害
- ログに「authentication worked 2 minutes after creation」パターン

**Phase to address:**
Phase 1（API統合）で実装。デプロイ自動化（Phase 3-4）では必須のエラーハンドリング。

### Pitfall 3: LLMコスト爆発（未キャッシュのコンテキスト繰り返し処理）

**What goes wrong:**
市場急変時に同じ市場データを数百回LLMに送信し、月間コストが予想の10-50倍に膨らむ。特に「なぜ動いたか」分析で、市場メタデータ（ルール、説明、履歴）を毎回フルコンテキストで送信してしまう。

**Why it happens:**
Prompt Cachingを使わずに、毎回の分析リクエストで市場の基本情報を再送信。Claude APIの場合、キャッシュ利用で90%コスト削減可能だが、実装しないと標準料金が適用される。リアルタイム分析の「速度優先」でキャッシュ設計を後回しにする心理的バイアス。

**How to avoid:**
1. 市場メタデータ（説明、ルール、過去データ）をPrompt Cachingの対象に設定
2. キャッシュヒット率を監視（目標: 80%以上）
3. 分析頻度に応じた段階的処理:
   - 急変検知: 軽量モデル（Haiku）で1行要約のみ
   - 詳細分析: リクエスト時のみ、Sonnetでキャッシュ活用
4. コスト上限アラート: 1日あたり$10を超えた時点で通知

**Warning signs:**
- 同一市場IDへのAPI呼び出しが1時間に10回以上
- LLMコストが日次で線形増加（キャッシュ効果なし）
- Cache hit rateが20%未満
- トークン消費量がユーザー数増加率を大幅に上回る

**Phase to address:**
Phase 2（AI分析機能）で必須実装。MVP時点でコスト構造を確立しないと、後の収益化が困難。

### Pitfall 4: Discord Rate Limit突破による通知ブラックアウト

**What goes wrong:**
市場急変時に大量の通知を短時間で送信し、Discordのrate limitに抵触。通知が最大10分間配信されず、ユーザーは重要なタイミングで情報を受け取れない。最悪の場合、IPレベルのCloudflareバンで一時的にサービス停止。

**Why it happens:**
Discordは1つのチャンネルで1秒あたり5メッセージ、グローバルで50リクエスト/秒の制限。予測市場は相関性が高く（例: 米大統領選関連市場が一斉に動く）、10-50市場が同時に急変すると通知が集中。開発者は「通知=即送信」と単純実装しがち。

**How to avoid:**
1. 通知キューイング: 優先度付きキュー（クジラ検知 > 急変 > 定期更新）
2. バッチング: 同一テーマの複数急変を1メッセージに統合（例: 「米政治市場3件が急変」）
3. Rate limit予測: 送信前に`X-RateLimit-Remaining`ヘッダーをチェック、0に近い場合はバッファリング
4. フォールバック: Webhookでの通知（channel messageより高いレート制限）
5. 通知間隔の動的調整: 高負荷時は5秒→30秒に自動延長

**Warning signs:**
- `429 Too Many Requests`エラーの発生
- `X-RateLimit-Remaining`が頻繁に1桁台
- 通知送信と実際の配信時刻のラグが30秒以上
- ユーザーからの「通知が遅れている」報告

**Phase to address:**
Phase 2（Discord通知）で優先度付きキュー実装。Phase 3（スケーリング）でバッチング強化。

### Pitfall 5: クジラ検知の誤検知地獄（市場性質の無視）

**What goes wrong:**
取引額の絶対値のみで「クジラ」判定し、小規模市場（流動性$1K以下）の$100取引や、Meme系市場の通常取引を「異常」として大量通知。ユーザーが通知をミュートし、本当の重要イベントも見逃す「オオカミ少年」状態に。

**Why it happens:**
Polymarket市場は流動性が$100万（大統領選）から$500（ニッチ市場）まで3桁以上の差がある。シンプルな閾値判定（例: $1K以上=クジラ）では、市場ごとの文脈を無視。また、市場タイプによる「通常の大口」が異なる（スポーツ賭博vs政治予測）。

**How to avoid:**
1. 市場流動性比率ベースの検知: 取引額が市場総流動性の5%以上
2. 市場カテゴリ別閾値:
   - 政治（高流動性）: $10K+
   - 暗号資産: $5K+
   - スポーツ/エンタメ: $2K+
   - その他: 流動性の10%
3. 時系列異常検知: その市場の過去7日平均取引額の3σ以上
4. False positive tracking: ユーザーの「不要な通知」フィードバックで閾値を調整

**Warning signs:**
- クジラ通知の70%以上が流動性$5K未満の市場
- 同一市場から1時間に5回以上のクジラ通知
- ユーザーのクジラ通知チャンネルミュート率が50%超
- 「通知が多すぎる」というフィードバック

**Phase to address:**
Phase 2（急変・クジラ検知）で文脈的閾値を実装。Phase 3でユーザーフィードバックによる自動調整。

### Pitfall 6: 日本語翻訳の文脈喪失（市場固有用語の誤訳）

**What goes wrong:**
「Will Trump win Pennsylvania?」を「トランプはペンシルバニアに勝ちますか？」と直訳し、予測市場文脈（選挙人獲得）が伝わらない。また、「YES/NO」を「はい/いいえ」と訳し、市場ポジションとしての意味が不明瞭に。

**Why it happens:**
汎用LLM翻訳は予測市場ドメインの暗黙知を持たない。「win」は「勝つ」だが、予測市場では「獲得する」「制する」がより正確。また、英語の市場タイトルは簡潔さ優先で文脈を省略するが、日本語では補完が必要。

**How to avoid:**
1. ドメイン特化翻訳辞書:
   - "win [state]" → "[州]を制する／獲得する"
   - "YES/NO" → "YES(実現) / NO(不実現)" または市場ごとにカスタム
   - "resolve" → "決済" (not "解決")
   - "whale" → "大口投資家" (not "クジラ")
2. 翻訳プロンプトに市場文脈を注入:
   ```
   これは予測市場のタイトルです。政治/スポーツ/経済の文脈で、
   日本語として自然かつ正確に翻訳してください。
   ```
3. 頻出市場タイプのテンプレート化（選挙、スポーツ結果、価格予測など）
4. 人間レビューフィードバックループ: 誤訳報告→辞書更新

**Warning signs:**
- ユーザーから「意味が分からない」フィードバック
- 同じ英語表現が複数の日本語訳で揺れる
- 市場タイトルをクリックしてPolymarketで原文確認する行動が頻発
- YES/NO市場で「どちらに賭けるべきか分からない」混乱

**Phase to address:**
Phase 1（データ表示）で翻訳辞書の初期実装。Phase 2-3でフィードバック反映。

### Pitfall 7: 「なぜ動いたか」分析のニュース不在問題

**What goes wrong:**
市場急変の理由分析で、LLMが「関連ニュースが見つかりません」や「推測: おそらく〇〇」と出力。ユーザーは「AIが適当なことを言っている」と不信感を持ち、プレミアム機能の価値が損なわれる。

**Why it happens:**
予測市場の急変は、(1)メジャーニュースが原因、(2)内部情報・ルーマーが原因、(3)クジラの投機的動き、の3パターン。ニュースAPIは(2)(3)をカバーできず、LLMは情報不足で推測に頼る。また、市場反応は速く（数分）、ニュースのインデックス遅延（5-15分）で因果関係を逃す。

**How to avoid:**
1. 多層ニュースソース:
   - 一般ニュースAPI（Reuters, Bloomberg）
   - X（旧Twitter）リアルタイム検索（市場キーワード監視）
   - Polymarket市場のコメント欄（内部情報の早期キャッチ）
2. 情報不足時の正直な表示:
   ```
   【分析中】明確な報道は未確認です。
   以下の可能性を調査中:
   - [仮説1]
   - [仮説2]
   ```
3. 人間エディターによる後追い分析（プレミアム機能）
4. 「分析不能」をメトリクスとして追跡（目標: 20%以下）

**Warning signs:**
- 分析の50%以上が「推測」「可能性」で始まる
- ユーザーが分析を読まずに市場に直接遷移
- プレミアム解約理由に「分析が役に立たない」
- 急変検知から分析出力まで10分以上のラグ

**Phase to address:**
Phase 2（AI分析）でニュースソース多様化。Phase 3で人間レビュー体制。

### Pitfall 8: フリーミアム境界線の曖昧さによる収益機会損失

**What goes wrong:**
無料ユーザーが「有料プランで何が得られるか」を理解できず、コンバージョン率が1%未満。または逆に、無料機能が過剰で有料プランの訴求力がない（"nice to have"止まり）。

**Why it happens:**
「無料: 遅延データ、有料: リアルタイム」は技術的には明確だが、ユーザーにとっての価値差が不明瞭（5分遅延でも十分？）。また、Discord通知の有無は価値が高いが、無料ユーザーは通知の存在自体を知らない。

**How to avoid:**
1. 価値ベースの境界設定:
   - **無料**: 朝のまとめダッシュボード（12時間遅延）、市場一覧、テーマ別グルーピング
   - **有料**: リアルタイム通知、急変アラート、AI分析、クジラ検知、30分以内のデータ更新
2. 無料ユーザーへのプレミアム体験の「チラ見せ」:
   - 「3時間前に急変通知がありました（有料プラン）」
   - 「この分析を読むにはプレミアムが必要です（最初の1行だけ表示）」
3. 使用量ベース制限: 無料で月10回の通知、超過後は有料提案
4. タイミング重視訴求: 「この市場は2時間前に急変しました。次回はリアルタイムで通知します」

**Warning signs:**
- 有料プラン詳細ページのバウンス率が70%以上
- 無料ユーザーの平均滞在日数が3日未満（価値が伝わらない）
- 有料登録ユーザーの半数が「試しに登録」で翌月解約
- サポート問い合わせ「無料と有料の違いが分からない」

**Phase to address:**
Phase 2-3でフリーミアム境界の実装と可視化。Phase 4でコンバージョン最適化（A/Bテスト）。

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| WebSocket再接続をライブラリに依存 | 実装時間1日短縮 | 20-30分ごとの接続断、ユーザー信頼喪失 | **Never** — MVP段階から必須実装 |
| すべての市場データをポーリング（WebSocket未使用） | 実装シンプル、デバッグ容易 | APIレート制限到達、5-10分の遅延 | **MVPのみ可** — 市場数50以下かつユーザー10人未満 |
| LLM翻訳をキャッシュなしで毎回実行 | 実装時間半日短縮 | コスト10倍、応答速度低下 | **Never** — 同じ市場は数時間-数日間表示される |
| Discord通知をキューなしで直接送信 | 実装30分、動作確認容易 | rate limit突破、重要通知の欠損 | **プロトタイプのみ** — ユーザー5人以下の検証段階 |
| クジラ検知を絶対額閾値のみで判定 | ロジック5行で完結 | 誤検知率80%、通知ミュート多発 | **初期MVP** — 1週間以内にコンテキスト判定へ移行 |
| すべてのエラーを汎用エラーページで処理 | エラーハンドリング1画面 | 原因特定に時間、ユーザー離脱増加 | **MVP初日のみ** — 週次で主要エラーを個別対応 |
| 市場データをリレーショナルDB正規化 | データ整合性完璧 | JOIN地獄、分析クエリが遅延 | **少数市場（100以下）かつ分析なし** — 拡張時にNoSQL併用 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| **Polymarket CLOB API** | API認証キー作成後に即座に使用 | 2-3分のwarm-up期間設定、指数バックオフリトライ実装 |
| **Polymarket WebSocket** | `readyState === OPEN`で接続正常と判断 | メッセージ受信頻度を監視、5分無音で再接続トリガー |
| **Discord Webhook** | チャンネルメッセージと同じレート制限と誤解 | Webhookは専用制限（より緩い）、高頻度通知時に優先使用 |
| **LLM API（Claude/OpenAI）** | 同じコンテキストを毎回フル送信 | Prompt Cachingで市場メタデータをキャッシュ、90%コスト削減 |
| **ニュースAPI** | 市場急変後に同期的にニュース取得 | ニュースを常時ポーリング＆保存、急変時は既存データから検索 |
| **市場データポーリング** | すべての市場を同じ頻度でポーリング | 流動性・活動度に応じた差別化（高流動性: 30秒、低流動性: 5分） |
| **Discord Bot認証** | Bot tokenをハードコード | 環境変数＋Secretsマネージャー、絶対にログ出力しない（CLOB SDK Issue #327参照） |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| **全市場データをメモリに保持** | サーバーメモリ使用率が時間とともに増加 | Redis/Memcachedでキャッシュ、TTL設定（市場データ: 1時間） | 市場数500以上またはサーバーメモリ2GB未満 |
| **LLM分析をリクエストごとに同期実行** | ダッシュボード読み込み10秒以上 | バックグラウンドジョブで事前分析、結果をキャッシュ | 同時ユーザー20人以上 |
| **Discord通知を市場変動ごとに個別送信** | 429エラー頻発、通知遅延10分 | 優先度付きキュー＋バッチング、同テーマは30秒以内に統合 | 1時間に通知100件以上 |
| **市場検索をRDBMS LIKE句で実行** | 検索レスポンス3秒以上 | Elasticsearch/Algoliaで全文検索、市場タイトル・説明をインデックス | 市場数1,000以上 |
| **すべてのAPIリクエストをログ出力** | ディスク使用量が日次10GB増加 | 重要イベント（エラー、急変検知）のみログ、デバッグログは開発環境限定 | リクエスト10万/日以上 |
| **WebSocketメッセージを1件ずつDB保存** | DB書き込み遅延、WebSocketバッファオーバーフロー | メモリバッファで10秒または100件ごとにバルク書き込み | メッセージ頻度100/秒以上 |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| **Polymarket API認証ヘッダーをエラーログに出力** | 認証情報漏洩（Issue #327）、不正取引 | ログ出力前にヘッダーをサニタイズ、`Authorization`を常にマスク |
| **Discord Bot tokenをGitリポジトリにコミット** | Bot乗っ取り、スパム送信でアカウントBAN | `.env`ファイル＋`.gitignore`、CI/CDはSecrets機能を使用 |
| **ユーザーの有料プラン状態をクライアント側で判定** | フロントエンド改ざんで無料ユーザーがプレミアム機能利用 | サーバー側で認証・認可、APIリクエストごとにプラン確認 |
| **市場データAPIをCORS制限なしで公開** | データスクレイピング、競合サービスへの転用 | 自ドメインのみ許可、APIキー必須、レート制限適用 |
| **Webhook URLをハードコード** | Discord Webhook乗っ取り、スパム送信 | 環境変数管理、Webhook署名検証（Discord側の署名機能利用） |
| **LLM分析プロンプトにユーザー入力を直接挿入** | プロンプトインジェクション、意図しない出力 | ユーザー入力をエスケープ、システムプロンプトと明確に分離 |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| **市場タイトルを英語のまま表示** | 日本語ユーザーが内容を理解できず離脱 | LLM翻訳＋ドメイン辞書で自然な日本語化、原文を副次表示 |
| **急変通知に市場リンクのみ** | 「なぜ通知が来たか」不明、通知をミュート | 通知内容を簡潔に説明（例: "トランプ勝率が55%→62%に急上昇"） |
| **無料・有料の違いが不明確** | コンバージョン率1%未満 | ダッシュボードに「有料プランなら3時間前に通知」を表示、価値を可視化 |
| **クジラ通知が多すぎる（誤検知）** | 重要通知を見逃す、通知をミュート | 市場文脈に応じた閾値、ユーザーが通知レベルを選択可能に |
| **AI分析が「推測」ばかり** | 分析を信頼せず、プレミアム解約 | 情報不足時は正直に表示（"調査中"）、確実な情報のみ提供 |
| **ダッシュボードが市場500件をフラット表示** | 情報過多で目的の市場を見つけられない | テーマ別グルーピング、流動性・注目度でソート、検索機能 |
| **朝のまとめが時系列順のみ** | 重要度が分からない、全部読む時間がない | 重要度順（流動性×変動率）、各テーマTop3のみ表示 |

## "Looks Done But Isn't" Checklist

- [ ] **Discord通知機能:** 通知送信コードが動作しても、Discordサーバーでのボット権限設定（`VIEW_CHANNEL`, `SEND_MESSAGES`）を忘れやすい — 実際のサーバーで送信テスト完了を確認
- [ ] **クジラ検知:** 絶対額閾値での検知は実装できても、市場ごとの流動性比率判定なしでは誤検知地獄 — 小規模市場（流動性$1K以下）でテスト
- [ ] **WebSocket接続:** 初回接続は成功しても、20-30分後の無音接続断ハンドリングがない — 1時間以上の連続動作テスト＋再接続ロジック確認
- [ ] **LLM翻訳:** 一般的な英文は翻訳できても、予測市場固有用語（resolve, YES/NO, win [state]）の文脈翻訳なし — ドメイン辞書の網羅性確認
- [ ] **フリーミアム境界:** 機能制限は実装済みでも、無料ユーザーへの「有料プランで得られるもの」の可視化なし — コンバージョンファネル測定
- [ ] **API認証エラーハンドリング:** 401エラーをキャッチしても、2分間の認証遅延を考慮したリトライロジックなし — キー作成直後のデプロイで検証
- [ ] **急変検知:** オッズ変動率の計算は正しくても、市場の流動性・カテゴリ・時間帯を無視した一律閾値 — 複数タイプの市場で精度測定
- [ ] **通知のレート制限対策:** キューイングは実装済みでも、Discord 429エラー後のRetry-Afterヘッダー読み取りなし — 高負荷時の挙動テスト

| **WebSocket接続断に気づかず数時間放置** | MEDIUM | 1. ヘルスチェックAPIで接続状態監視追加（30分）、2. アラート設定（15分）、3. 自動再起動スクリプト（1時間） |
| **LLMコストが予算の10倍** | HIGH | 1. 緊急でPrompt Caching実装（4時間）、2. 一時的に分析頻度を1/10に削減（1時間）、3. コスト監視ダッシュボード構築（2日） |

| **クジラ通知誤検知でユーザー離反** | MEDIUM | 1. 流動性比率ベース判定に移行（2時間）、2. 既存ユーザーに改善を通知・信頼回復（1日）、3. 誤検知率をダッシュボード化（半日） |
